## 二、注释规范定义

### 2.1 文件级注释

每个 `.ts` / `.vue` 文件顶部必须包含：

```typescript
/**
 * @file 文件名.ts
 * @description 该文件的核心职责（一句话）
 * @module 所属模块（如 features/magnetSearch）
 */
```

### 2.2 函数/方法级注释

公共函数和复杂私有函数必须包含：

```typescript
/**
 * 从多个磁力源并发搜索并合并结果
 *
 * @param keyword - 搜索关键词（番号）
 * @param sources - 要查询的磁力源列表
 * @param options - 并发控制选项
 * @returns 合并去重后的磁力链接数组
 * @throws {NetworkError} 当所有源都超时时
 *
 * @example
 * const results = await searchMagnet('ABC-123', ['sukebei', 'nyaa'], { timeout: 5000 })
 */
```

### 2.3 行内注释

- 解释 **为什么** 这样做，而非 **做了什么**
- 复杂业务逻辑、hack、workaround 必须标注原因
- 引用外部规范/issue 时附上链接

```typescript
// ✅ 好的注释
// Chrome 110+ 限制 Service Worker 最大生命周期为 5 分钟，需用 alarm 续期
// 参考：https://developer.chrome.com/docs/extensions/develop/concepts/service-workers/lifecycle

// ❌ 不好的注释
// 设置 alarm
chrome.alarms.create(...)
```

### 2.4 Vue 组件注释

```vue
<script setup lang="ts">
/**
 * 磁力搜索结果列表组件
 * 负责渲染搜索结果、处理分页、支持推送到115网盘
 * @emits push - 推送到网盘
 * @emits copy - 复制磁力链接
 */
</script>
```

### 2.5 类型定义注释（行尾风格）

**核心原则：只注释非自明字段，自解释字段（`id`、`name`、`createdAt` 等）不加注释。**

```typescript
/** 演员记录 —— IndexedDB `actors` 表的完整数据结构 */
export interface ActorRecord {
  id: string;
  name: string;
  aliases: string[];                                  // 搜索匹配用的别名
  gender: 'male' | 'female' | 'unknown';
  category: 'unknown' | string;                       // 演员分类标签
  avatarUrl?: string;
  profileUrl: string;
  createdAt: number;                                  // Unix 时间戳（毫秒）
  blacklisted?: boolean;                              // 黑名单演员不显示增强信息
  manuallyEditedFields?: string[];                    // 同步时不会覆盖这些字段
  wikiData?: {                                        // 演员元数据刷新功能获取
    age?: number;
    heightCm?: number;
    ig?: string;                                      // Instagram
    tw?: string;                                      // Twitter / X
    source?: 'wikipedia' | 'xslist';
    fetchedAt?: number;
  };
}
```

**注释密度参考：**
- ✅ 有业务含义的字段（`blacklisted`、`manuallyEditedFields`）→ 必须注释
- ✅ 容易误解的字段（`category`、`source` 的枚举含义）→ 必须注释
- ✅ 单位/格式不明确的字段（时间戳是秒还是毫秒）→ 必须注释
- ✅ 嵌套对象（`syncInfo`、`wikiData`）→ 注释用途，内部字段按需
- ❌ 自解释字段（`id`、`name`、`email`、`title`）→ 不加注释
- ❌ 通用类型字段（`createdAt`、`updatedAt`、`enabled`）→ 除非有特殊含义

## 四、自动化测试注释评估

### 4.1 结论：**需要补全，但优先级低于源码**

### 4.2 理由

| 维度 | 分析 |
|------|------|
| **协作必要性** | 测试是理解业务逻辑的重要参考，新人通过测试了解预期行为 |
| **维护成本** | 测试文件 ~160 个，与源码同步修改时需理解上下文 |
| **当前状况** | 部分测试文件已有简要描述，但缺少对测试场景和边界条件的说明 |
| **投入产出比** | 测试注释的 ROI 低于源码，但不可忽略 |

### 4.3 测试注释规范

```typescript
describe('TaskScheduler', () => {
  describe('任务优先级排序', () => {
    it('应按 priority 降序排列，同优先级按创建时间升序', () => {
      // Arrange: 创建3个不同优先级的任务
      // Act: 调用排序函数
      // Assert: 验证排序结果
    })

    it('应将已暂停的任务排到最后', () => {
      // 场景：混合 running/paused/pending 任务
    })
  })
})
```

## 五、执行策略

### 5.1 分工建议

- **Phase 1（类型+共享）**：由熟悉项目整体架构的核心成员负责
- **Phase 2（平台层）**：由负责基础设施的成员负责
- **Phase 3（features/）**：各功能模块 Owner 负责自己模块
- **Phase 4（后台+UI）**：前后端分别负责
- **Phase 5（脚本）**：构建负责人顺手完成

### 5.2 质量保障

- PR Review 时检查注释是否符合规范
- 每个 Phase 完成后更新此文档的进度状态
- 可考虑引入 `eslint-plugin-jsdoc` 自动检查注释完整性

| 范围 | 工作量 |
|------|--------|
| 源码注释（Phase 1-5） | **约 15-18 人天** |
| 测试注释（T1-T5） | **约 5 人天** |
