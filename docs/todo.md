## 版本号分级规则

### Patch：`MAJOR.MINOR.PATCH`

适用于兼容性修复、稳定性收口、小范围体验优化和已上线功能补齐。

- 修复测试、构建、发布说明、回归问题
- 修复 WebDAV、Emby/Jellyfin、番号库、磁力等已发布能力的问题
- 小范围 UI 文案、提示、选择器校准、设置项默认值调整
- 不改变用户数据结构的轻量功能补齐

发布判断：同一具体 Patch 版本的 P0 阻断项清零，P1 收口项完成，验证链路通过。不要因为单个小优化单独发版，Patch 也应是一组同主题修复/体验收口。

### Minor：`MAJOR.MINOR.0`

适用于完整新能力、跨模块工作流、可感知的功能组合和中等架构调整。

- 新增一组完整用户能力，并且用户能感知到一条完整工作流
- 需要新增设置项、数据流、后台任务、页面入口和测试覆盖
- 影响多个功能域，但保持旧配置、旧数据、旧入口兼容
- 架构调整服务于一个明确功能主题
- 同一功能需要拆成多个 Patch 铺垫时，最后一个整合版本才考虑升 Minor

发布判断：连续多个 Patch 已经完成铺垫，并形成一组完整新能力时，再进入下一个 Minor。

### Major：`2.0.0`

适用于不兼容变更、存储边界重构、权限模型变化和旧入口移除。

- 移除旧兼容路径、旧配置入口或旧数据结构
- 改变用户数据导入、备份、恢复、同步的兼容规则
- 引入需要用户重新授权或重新配置的权限变化
- 重做核心导航、设置结构或跨页面工作流

发布判断：需要迁移说明、回滚策略、数据备份提醒和跨版本升级测试。

## 版本内优先级

- P0：发布阻断。测试失败、构建失败、数据风险、迁移风险、发布说明缺失。
- P1：版本核心。这个版本承诺交付的主线功能或主线收口。
- P2：随版增强。低风险体验优化、文案、提示、轻量 UI、局部校准。
- P3：评估池。需要调研、拆分设计或等待稳定周期的事项。
